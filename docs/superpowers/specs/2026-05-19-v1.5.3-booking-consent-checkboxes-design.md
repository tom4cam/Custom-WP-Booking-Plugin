# v1.5.3 — Required email/SMS consent checkboxes — Design

**Date:** 2026-05-19
**Status:** Approved
**Scope:** Add two required consent checkboxes (email + SMS) to the booking
form, the post-booking quick-register form, and the standalone account-page
register form. Make phone required everywhere. Persist consent on the
booking row and on the WP user record. Wording is settings-configurable.

## Goal

1. Capture explicit, TCPA-style consent for transactional emails and SMS
   at every point where contact info is collected. Both checkboxes must be
   ticked to submit; both must be confirmed server-side.
2. Persist consent values + timestamp on the booking row (per-event proof)
   **and** on the WP user account (per-account proof), so that the audit
   trail survives even if all bookings are purged.
3. Keep the wording editable in admin settings so legal/compliance copy
   can be updated without a code change.

## Non-goals

- Account-page consent-management UI (no toggle for already-registered
  users to update consent). Out of scope for v1.5.3.
- Marketing-email consent. Only transactional (confirmation + reminder)
  consent is captured. Same with SMS.
- Custom field-level error styling beyond the existing
  `.caswell-error` pattern.
- Backfilling consent for existing rows. Existing bookings get
  default `0` — they pre-date the requirement.

## Wording — default copy

Both lines use `{business_name}` which is replaced at render time with
`caswell_get_option( 'business_name', get_bloginfo( 'name' ) )`.

- **Email** (`email_consent_text` default):
  `I agree to receive appointment confirmation and reminder emails from {business_name} at the email address provided.`
- **SMS** (`sms_consent_text` default):
  `I agree to receive appointment confirmation and reminder text messages from {business_name} at the phone number provided. Message frequency varies. Msg & data rates may apply. Reply STOP to opt out.`

Both settings render in **Admin → Notifications** as `textarea` fields
with the placeholder hint `{business_name}` documented inline.

## DB migration

Three new columns on `{$wpdb->prefix}caswell_bookings`:

| Column | Type | Default |
|---|---|---|
| `email_consent` | `TINYINT(1) NOT NULL` | `0` |
| `sms_consent` | `TINYINT(1) NOT NULL` | `0` |
| `consent_timestamp` | `DATETIME NULL` | `NULL` |

`Caswell_Booking_DB::BOOKINGS_VERSION` bumps from `4` → `5`. The plugin's
`plugins_loaded` hook compares `caswell_db_version < BOOKINGS_VERSION` and
calls `create_tables()` → `dbDelta` runs the migration idempotently.

`Caswell_Booking_DB::insert_booking()` `$defaults` array gains:

```php
'email_consent'     => 0,
'sms_consent'       => 0,
'consent_timestamp' => null,
```

No new indexes — these columns aren't queried.

## User meta

On registration, the three values are stored as WP user meta keyed:

- `_caswell_email_consent` — `'1'` or `'0'`
- `_caswell_sms_consent` — `'1'` or `'0'`
- `_caswell_consent_timestamp` — `current_time( 'mysql' )` at registration

Convention: `_caswell_` user-meta prefix (per CLAUDE.md).

JS (`public/js/booking.js` line 105–117) — prefill phone from the booking
form, include consent flags in the payload, validate client-side:

```js
$(document).on('submit', '#caswell-quick-register', function (e) {
    e.preventDefault();
    const $form = $(this);
    const phone = $.trim($form.find('[name="phone"]').val() || $('#caswell-phone').val());
    const emailConsent = $form.find('[name="email_consent"]').is(':checked');
    const smsConsent   = $form.find('[name="sms_consent"]').is(':checked');
    if (!phone || !emailConsent || !smsConsent) {
        $form.prepend('<p class="caswell-error">Please fill in phone and confirm both consent checkboxes.</p>');
        return;
    }
    ajaxRequest('caswell_register', {
        name:           $('#caswell-name').val(),
        email:          $form.find('[name="email"]').val(),
        phone:          phone,
        password:       $form.find('[name="password"]').val(),
        email_consent:  emailConsent ? 1 : 0,
        sms_consent:    smsConsent   ? 1 : 0,
    }, /* success */ /* error */);
});
```

Checkboxes are **prefilled checked** because the user just ticked the same
boxes a moment ago to book. They can untick if they want — and if they do,
client-side validation blocks submission with a clear message.

## New helper

`caswell-booking.php` next to the other `caswell_*` helpers:

```php
/**
 * Render a consent line with {business_name} interpolated.
 *
 * @param string $kind 'email' or 'sms'.
 * @return string
 */
function caswell_render_consent_text( $kind ) {
    $defaults = [
        'email' => 'I agree to receive appointment confirmation and reminder emails from {business_name} at the email address provided.',
        'sms'   => 'I agree to receive appointment confirmation and reminder text messages from {business_name} at the phone number provided. Message frequency varies. Msg & data rates may apply. Reply STOP to opt out.',
    ];
    $key      = "{$kind}_consent_text";
    $template = caswell_get_option( $key, $defaults[ $kind ] ?? '' );
    $business = caswell_get_option( 'business_name', get_bloginfo( 'name' ) );
    return str_replace( '{business_name}', $business, $template );
}
```

Pure function — easy to unit-test with Brain Monkey.

## Migration / compatibility

- **Schema**: existing rows get `email_consent=0`, `sms_consent=0`,
  `consent_timestamp=NULL`. Acceptable — they pre-date the requirement
  and Ryan's not required to retroactively prove consent for past
  bookings.
- **Existing users**: no consent on file. They keep working — the new
  fields only gate *new* bookings/registrations. Past bookings remain
  bookable for reschedule via signed link (which skips the checkboxes).
- **Settings**: defaults render correctly with no admin action.
- **Reschedule link flow**: server inherits consent from the original
  booking when the form was loaded via signed link.

## Verification

Manual smoke test on the live site after Dropbox sync:

- **New booking happy path**: open booking page → phone field shows
  `*` and is required → submit blocked until both checkboxes ticked →
  successful booking → row in DB has `email_consent=1`, `sms_consent=1`,
  `consent_timestamp` set.
- **Submit with unchecked**: untick either checkbox → submit blocked with
  the inline error. Server-side: hit the AJAX endpoint with curl bypassing
  JS → `wp_send_json_error` returns the consent error.
- **Phone required**: leave phone empty → submit blocked client-side and
  server-side.
- **Reschedule flow**: open form via reschedule link → consent checkboxes
  hidden, phone still required → successful reschedule → new booking row
  has consent flags + timestamp copied from the original.
- **Quick register (step 4)**: complete a booking → step 4 register form
  shows phone (prefilled from booking) and consent checkboxes (prefilled
  checked) → submit → new user created → user meta has all three consent
  keys + `caswell_phone`.
- **Main register**: log out → account page → register tab → phone field
  visible and required → checkboxes required → successful registration →
  user meta populated.
- **Wording renders**: confirm `{business_name}` is replaced (not shown
  literally) on every form. Test with `business_name` setting empty (site
  name fallback) and set (override).
- **Logs**: `caswell_log('booking', 'Consent recorded', ...)` and
  `caswell_log('auth', 'Registration consent recorded', ...)` appear in
  the standard log location.
